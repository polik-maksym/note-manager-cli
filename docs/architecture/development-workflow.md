# Development Workflow

## Мета

Цей документ описує процес розробки, якого ми дотримуємося у проєкті
**note-manager-cli**.

Основна ідея:

> **Спочатку думаємо → потім проєктуємо → лише після цього пишемо код.**

------------------------------------------------------------------------

# 1. Requirements (Вимоги)

Перед написанням коду сформулюйте вимоги.

Вимоги відповідають на питання:

> **Що повинна робити функція?**

Приклад:

-   пошук нотаток за тегом;
-   пошук без урахування регістру;
-   показуються лише знайдені нотатки;
-   режим лише для читання.

------------------------------------------------------------------------

# 2. Architecture (Архітектура)

Після визначення вимог рішення розділяється на логічні частини.

> **На які функції або модулі поділити рішення?**

Приклад:

``` text
search_by_tag()
        │
        ├── load_notes()
        ├── find_notes_by_tag()
        └── print_note()
```

------------------------------------------------------------------------

# 3. Activity Diagram (Алгоритм)

Перед реалізацією створюється Activity Diagram.

Вона відповідає на питання:

> **У якій послідовності виконується алгоритм?**

Файли діаграм:

``` text
docs/diagrams/activity/
```

Рекомендовані формати:

-   \*.drawio --- редагування;
-   \*.svg --- перегляд.

------------------------------------------------------------------------

# 4. Design Review

Перед написанням коду перевіряються:

-   вимоги;
-   архітектура;
-   алгоритм;
-   простота рішення;
-   можливість розвитку.

------------------------------------------------------------------------

# 5. Implementation (Реалізація)

Після Design Review реалізується Python-код.

------------------------------------------------------------------------

# 6. Code Review

Перевіряються:

-   читабельність;
-   дублювання;
-   стиль;
-   типізація;
-   відповідність архітектурі.

------------------------------------------------------------------------

------------------------------------------------------------------------

# Основні принципи

-   Спочатку аналіз, потім код.
-   Простота важливіша за складність.
-   Документуємо важливі рішення.
-   Одна функція --- одна відповідальність.
-   Діаграми є частиною документації.

> **Код --- це результат добре продуманого процесу.**
