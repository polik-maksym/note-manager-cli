# Architecture Decisions

**Проєкт:** Note Manager CLI

**Версія документа:** 1.0

**Останнє оновлення:** 2026-06-26

------------------------------------------------------------------------

Цей документ містить важливі архітектурні рішення, прийняті під час
розробки проєкту **note-manager-cli**.

Мета документа --- фіксувати не тільки *що* було реалізовано, а й *чому*
було прийнято саме таке рішення.

------------------------------------------------------------------------

## ADR-001. Користувач працює з порядковими номерами замість ID

### Статус

**Accepted**

### Контекст

Кожна нотатка має внутрішній унікальний `id`, який використовується для
ідентифікації запису в JSON-файлі.

### Рішення

У CLI користувач працює лише з порядковими номерами нотаток.

### Причини

-   простіше вводити номер;
-   інтерфейс зрозуміліший;
-   `id` залишається внутрішньою деталлю реалізації.

### Наслідки

-   використовується `enumerate(notes, start=1)`;
-   існує функція `get_note_by_number()`;
-   у майбутньому `id` можна використати для API або експорту.

### Прийнято

2026-06-26

## ADR-002. Виведення нотатки винесено в `print_note()`

### Статус

**Accepted**

### Контекст

Код виведення повторювався б у кількох функціях.

### Рішення

Створити окрему функцію `print_note(note)`.

### Причини

-   менше дублювання;
-   одне місце для зміни формату;
-   простіші `view_note()` та `edit_note()`.

### Наслідки

`print_note()` знаходиться в `notes.py`, а не в `utils.py`.

### Прийнято

2026-06-26

## ADR-003. Вибір нотатки винесено в `select_note()`

### Статус

**Accepted**

### Контекст

Функції `view_note()`, `delete_note()` та `edit_note()` мали однаковий
сценарій вибору нотатки.

### Рішення

Створити `select_note()` без аргументів.

Функція повертає:

``` python
notes, note, number
```

### Причини

-   універсальний текст «Вкажіть номер нотатки:» достатній;
-   усунуто дублювання коду;
-   функції стали коротшими.

### Наслідки

`view_note()` використовує тільки потрібне значення через `_`, а
`delete_note()` і `edit_note()` отримують ще `notes` і `number`.

### Прийнято

2026-06-26

## ADR-004. Одна Git-гілка --- одна логічна задача

### Статус

**Accepted**

### Рішення

Кожна окрема задача реалізується в окремій Git-гілці.

Приклади:

-   `feature/view-note`
-   `feature/delete-note`
-   `feature/edit-note`
-   `docs/architecture-decisions`
-   `refactor/select-note`

### Причини

-   зрозуміла історія Git;
-   прості Pull Request;
-   ізольовані зміни.

### Прийнято

2026-06-26

------------------------------------------------------------------------

## ADR-005. ID не відображається користувачу

### Статус

**Accepted**

### Рішення

`id` залишається внутрішнім ідентифікатором і не показується у CLI.

### Причини

-   користувачу він не потрібний;
-   інтерфейс стає чистішим;
-   поле залишається доступним для майбутніх можливостей.

### Прийнято

2026-06-26

------------------------------------------------------------------------

## ADR-006. `utils.py` не знає про структуру нотатки

### Статус

**Accepted**

### Рішення

Універсальні функції залишаються в `utils.py`, а функції, що працюють із
нотаткою, --- у `notes.py`.

### Причини

-   чіткий розподіл відповідальності;
-   модулі менше залежать один від одного.

### Прийнято

2026-06-26
